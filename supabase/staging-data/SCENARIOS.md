# Staging seed data — scenarios

This file is the **plan**, not the seed. It enumerates which accounts, insyts,
purchases, and storage fixtures the staging environment should contain after a
fresh seed run, and why each one exists. The SQL files and the asset-upload
script that implement the plan live in this same folder and are built **only
after this doc is reviewed**.

## Why a dedicated folder

Past test data has been smuggled in via migrations (e.g.
`20260512030000_seed_test_gallery_images.sql`). That mixes one-off staging
seed with schema history and is hard to refresh. This folder:

- Lives next to `migrations/` and `functions/` inside the `supabase/` repo so
  it ships with the same git push that ships schema.
- Is invoked explicitly (`./seed.sh` or `deno run seed.ts`), never on every
  migration apply.
- Is **staging-only**. The seed script refuses to run when the project ref
  matches the production ref (`krapqgxrqprtajatxjzd`).

## Idempotency contract

A re-seed run produces the exact same logical end state without touching
anything outside the seed's scope. Every reset distinguishes seeded data
from non-seeded data by a **deterministic marker baked into the row**, not
by user email — so an anchor's hand-created insyts / purchases / social
links survive a reset, while their seed-owned ones get rewritten.

### Seed markers (the scope of every DELETE in the reset)

| Resource                 | Marker the reset filters by                                       |
| ------------------------ | ----------------------------------------------------------------- |
| `public.insyts`          | `insyt_id LIKE 'seed-insyt-%'` (slug pattern)                     |
| `public.insyt_attachments` | CASCADEs from `insyts` delete; nothing else.                    |
| `public.purchases`       | `stripe_session_id LIKE 'seed-sess-%'`                            |
| `public.insyt_status`    | CASCADEs from `insyts` delete.                                    |
| `public.user_social_links` for anchors | UPSERT by `(user_id, platform)` — never deleted, only overwritten platform-by-platform |
| `public.user_social_links` for non-anchors | CASCADEs from the auth.users drop.                    |
| `public.agreement_acceptances` | scoped by user — non-anchors are dropped with the user; anchor's rows are UPSERTed by `(user_id, agreement_version_id)` |
| Storage `creator-avatars` / `creator-covers` | path prefix matches a seeded user's `auth_user_id`    |
| Storage `insyt-thumbnails` / `insyt-images` / `insyt-pdfs` | path prefix matches a `seed-insyt-` slug |
| Webflow CMS Creators     | item lookup by `webflow_creator_id` stored on `public.users`; anchors PATCH, non-anchors DELETE+POST |
| Webflow CMS Insyts       | `env = "staging"` field + slug `LIKE 'seed-insyt-%'`              |

### Per-resource behavior

1. **Anchor accounts** (`lukas.lampe@…`, `lukaslampe87@gmail.com`)
   - `auth.users.id` UUID, email, password, email-confirmed flag,
     `webflow_creator_id` reference — **all preserved**.
   - `public.users` columns the seed owns are **overwritten on every run**:
     `display_name`, `bio`, `headline`, `location`, `website`, `username`,
     `profile_image_url`, `cover_image_url`, `is_creator`,
     `creator_terms_accepted_at`. Columns the seed does **not** own
     (`stripe_connect_onboarded`, `total_earnings_usd`, `last_logged_in_at`,
     `follower_count`, anything app-mutable that the seed's not modeling)
     stay untouched.
   - Hand-tweaking anchor `bio` between runs: gets reverted on next reset.
     That's intentional — tests need a known anchor bio.

2. **Non-anchor test accounts** (`seed-*@getinsyts.test`): dropped by
   email + recreated. UUIDs change every run. Anything FK-referenced by
   their user id CASCADEs away with them.

3. **Insyts**: every row where `insyt_id LIKE 'seed-insyt-%'` is
   `DELETE`d before the new insert. Real insyts the anchor created
   manually (whose insyt_id was generated by `submit-create-insyt` and
   doesn't match the pattern) are **invisible to the reset**.

4. **Purchases**: every row where `stripe_session_id LIKE 'seed-sess-%'`
   is wiped. Real test-mode Stripe purchases (with actual session ids
   like `cs_test_…`) are invisible to the reset.

5. **Anchor social links**: UPSERT per platform. If the anchor manually
   added a fifth social platform the seed doesn't manage, it survives.
   If the seed lists `{youtube, instagram, facebook, tiktok}` for the
   anchor and the anchor manually added `youtube`, the seed overwrites
   that one platform's handle. No `DELETE FROM user_social_links WHERE
   user_id = anchor` — that would destroy survivable rows.

6. **Storage objects under `<seeded-user-uuid>/…` for anchor avatar +
   cover paths** are removed and re-uploaded with the fixture SVG. The
   wider bucket is untouched.

7. **Webflow CMS** items are PATCHed to match Supabase, in both
   directions of the schema:
   - Creators: call the existing `sync-creator-to-webflow` edge function
     once per seeded creator.
   - Insyts: the seed talks to the Webflow API directly (POST/PATCH on the
     Insyts collection), **bypassing** `submit-create-insyt` → n8n. n8n
     also creates Stripe products and posts to other side channels;
     none of that should fire for seed data. Webflow item ids are written
     back into `public.insyts.webflow_item_id` so re-seeds can PATCH (not
     duplicate).
   - Teardown: before deleting Supabase rows for seeded users / insyts,
     the seed reads their `webflow_creator_id` / `webflow_item_id`,
     deletes the matching Webflow CMS items, and only then drops the
     Supabase rows. Otherwise re-seeds leave orphan Webflow items behind
     and the public site shows ghost rows.

Anything a tester or test run wrote that doesn't fall into the categories
above survives the re-seed.

### UI-driven-start vs seed-as-end fixtures

Most personas above are **end states** — the seed puts them in the shape
the test wants to inspect, and the test never has to click them into that
shape. That keeps tests fast.

Two personas in this folder exist specifically as **starting points for
UI-driven flow tests** (where the point of the test is the click-through
itself, not the end state):

| Email                                      | Starts as                  | Test that promotes it                                             |
| ------------------------------------------ | -------------------------- | ----------------------------------------------------------------- |

on every run, so the UI test always starts from a known pre-flow state. The
test itself is what mutates them — and any data it writes (a new
`purchases` row, an `agreement_acceptances` row, `is_creator=true`) is
allowed to survive until the next re-seed, when it's wiped clean.

## Storage / asset fixtures

Six small files vendored under `staging-data/fixtures/` are reused across
every seeded user and insyt. They render distinct-looking placeholders so
QA can tell them apart at a glance, but they are NOT photographic content —
they are clearly labeled "seed" so nothing on staging looks like real user
data.

| Fixture file              | Format | Size  | Used as                                                     |
| ------------------------- | ------ | ----- | ----------------------------------------------------------- |
| `avatar.svg`              | SVG    | <1 KB | All seeded avatars (anchor + full + long-strings + stripe-not-onboarded) |
| `cover.svg`               | SVG    | 1 KB  | All seeded covers (anchor + full)                           |
| `thumbnail.svg`           | SVG    | 1 KB  | Every seeded insyt thumbnail (insyts #1-5, #7-15 — all except #6 which is deliberately missing) |
| `gallery-wide.svg`        | SVG    | 1 KB  | Gallery slots in landscape aspect ratio                     |
| `gallery-tall.svg`        | SVG    | 1 KB  | Gallery slots in portrait aspect ratio                      |
| `sample.pdf`              | PDF    | <1 KB | Every seeded PDF attachment (insyts #2, #5, #14)            |

Total committed weight: ~5 KB.

The seed script uploads each fixture once per target storage path (so the
*same* SVG bytes live at many object keys, e.g.
`creator-avatars/<auth_user_id>/avatar-seed.svg` for every seeded
creator). SVGs render natively in Webflow's `<img>` bindings, so no
conversion to PNG/JPG is needed at upload time.

### Bucket paths the seed enforces

| Bucket               | Path convention                          | Fixture reused                              |
| -------------------- | ---------------------------------------- | ------------------------------------------- |
| `creator-avatars`    | `<auth_user_id>/avatar-seed.svg`         | `avatar.svg`                                |
| `creator-covers`     | `<auth_user_id>/cover-seed.svg`          | `cover.svg`                                 |
| `insyt-thumbnails`   | `<insyt_id>.svg`                         | `thumbnail.svg`                             |
| `insyt-images`       | `<insyt_id>/<position>.svg`              | `gallery-wide.svg` or `gallery-tall.svg` (alternating by position so each gallery has mixed aspect ratios) |
| `insyt-pdfs`         | `<insyt_id>/<filename>.pdf`              | `sample.pdf`                                |
| `creator-agreements` | (untouched by seed)                      | Pre-existing canonical agreement PDF        |

### Why SVG + one PDF

- **Deterministic by definition.** The fixture file IS the seeded image —
  no per-run randomness from Picsum, no flaky CDN fetch.
- **Tiny repo footprint.** All six fixtures together are ~5 KB. Compare
  with ~3-5 MB if we vendored real JPGs for the same coverage.
- **No network on seed runs.** A fresh `git clone` + `./seed.ts` works
  offline. Useful for CI, useful when working on a plane.
- **Clearly marked as fake.** The SVGs literally have "seed avatar",
  "seed cover", "seed thumbnail" baked into the visible text, so no one
  mistakes a screenshot of the staging site for production data.
- **Webflow CMS Image fields accept SVG.** Verified against the existing
  `profile-image` binding (it just renders whatever URL it gets).

If a future test specifically needs photographic content (e.g. visual
regression testing of the gallery layout against real photos), add the
needed image to `fixtures/` and update the lookup table — the seed
script's path-to-fixture map is a single config object.

## Webflow CMS sync

Supabase is the source of truth for both creators and insyts. The Webflow
CMS is a denormalised read cache that the Webflow templates render from.
After the seed, **both** must match Supabase, or the public pages show
stale / missing data. Seeded re-runs must leave the Webflow CMS in a
clean state.

### Creators collection (id `6a0ae123ed5b01902ba61136`)

Already wired: the seed invokes the existing
`sync-creator-to-webflow` edge function once per seeded creator (the same
function the in-product Edit Profile modal and avatar upload use). That
function does an upsert + publish via Webflow's `/live` endpoint and
writes the resulting item id back to `public.users.webflow_creator_id`.
Re-seeds that drop and recreate non-anchor creators DELETE the
corresponding Webflow item first, then let `sync-creator-to-webflow`
re-create it.

The anchor creator (`lukas.lampe@…`) keeps the same Webflow Creator id
(`6a0b011b41ec37704361e22f`) across runs — only its field data refreshes.

### Insyts collection (id `69d23ea82129dff2b2dc134d`)

The Insyts collection schema is:

| Webflow slug          | Webflow type   | Source in Supabase                              |
| --------------------- | -------------- | ----------------------------------------------- |
| `name` (required)     | PlainText      | `insyts.title`                                  |
| `slug` (required)     | PlainText      | `insyts.insyt_id` (deterministic seed slug)     |
| `abstract`            | PlainText      | `insyts.abstract`                               |
| `price`               | PlainText      | `insyts.price_eur` formatted as e.g. `"19.00"`  |
| `stripe-payment-link` | Link           | `insyts.stripe_payment_link_url` (placeholder for seeded paid insyts; null for free ones) |
| `tags`                | PlainText      | `array_to_string(insyts.tags, ',')`             |
| `creator-email`       | Email          | `insyts.creator_email`                          |
| `sport`               | PlainText      | `insyts.sport`                                  |
| `hidden`              | Switch         | `insyts.is_hidden`                              |
| `env`                 | PlainText      | literal `"staging"` for seeded items so prod tests can filter them out |
| `read-time-min`       | Number         | `insyts.read_time_min`                          |
| `thumbnail-url-2`     | PlainText      | resolved public URL for `insyts.thumbnail_url` (the `insyt-thumbnails` bucket) |
| `creator`             | Reference → Creators | The creator's `webflow_creator_id` (looked up after the Creator sync runs) |

The seed flow for Insyts is therefore:

1. **Run the Creator sync first** so every seeded creator has a
   `webflow_creator_id` available (the Insyt rows need to reference it).
2. **For each seeded insyt** (`public.insyts.env = 'staging'`):
   - If `insyts.webflow_item_id` is set: `DELETE` the Webflow item, then
     re-create. (Simpler than reconciling and avoids field drift —
     re-seeds are full rebuilds for non-anchor data.)
   - `POST /collections/<insyts-collection-id>/items/live` with the
     `fieldData` shape above.
   - Write the new Webflow item id back to `insyts.webflow_item_id` and
     publish.
3. **Sweep**: list all Webflow Insyts items whose `env = "staging"` and
   whose `slug` doesn't match any seeded insyt — those are orphans from a
   previous seed and get deleted. Same sweep for Creators (by
   `auth-user-id` matching the seeded list).

Status-quo insyts in Webflow that don't have `env = "staging"` are left
alone — that's how we avoid clobbering anything a human created in the
Designer.

### Why bypass `submit-create-insyt` + n8n for the seed

`submit-create-insyt` forwards the payload to an n8n webhook
(`N8N_CREATE_INSYT_URL`) which is responsible for the full publish chain:
Stripe Product + Price creation, Stripe Payment Link creation, Webflow
Insyts item creation, and a couple of notifications. For seeded fake data
none of that should fire — we don't want fake Stripe products in the
account, and we don't want notification noise on every re-seed. So:

- Seed talks to Webflow API directly for Insyts items.
- `stripe_product_id` / `stripe_price_id` / `stripe_payment_link_url` on
  seeded insyts are placeholder strings. Tests that need real Stripe
  state wire it up in the test itself; see the "Out of scope" section.

### Re-seed safety

Both syncs assume the Webflow API token configured for the seed has
**Site write** scope on the staging site only. Production safety is
two-layer: (1) the seed refuses to run when `--project-ref` is the prod
Supabase ref, and (2) the seed only ever writes Insyts items with
`env = "staging"`, so even if it somehow ran against a prod-pointed
Webflow site, the orphan sweep wouldn't touch human-created items.

`WEBFLOW_API_TOKEN` for the seed lives in the same `staging-data/.env`
as the Supabase service-role key — it is **not** the token shared with
the edge functions (those use Supabase Edge Function secrets); the seed
needs its own copy because it runs from the dev's machine. Treat it
like any other secret: gitignore, rotate if leaked,
[[reference-webflow-api-token]] explains where to mint it.

### `.env` files

```
supabase/supabase/staging-data/
├── .env.example   committed; shows the keys with empty values
└── .env           gitignored; each dev fills in their copy
```

`.env.example` ships with the doc; the actual `.env` is created by the
developer following the README. The seed script bails out with a clear
error if any required variable is missing, rather than reading from a
fallback.

### What the values *should not* be

- Not in `~/.zshrc` / `~/.bashrc` exported globally — leaks into every
  shell.
- Not pasted into git commits, even briefly. If a key ends up in history,
  rotate it immediately in the Supabase dashboard (Project Settings → API
  → "Reset service_role key") — old commits keep the leaked value
  forever.
- Not committed to the inner Supabase repo's `.env` patterns either —
  the supabase-deploy skill already calls out that secrets belong in the
  dashboard, not the CLI, but the seed-script case is the one exception
  where a local `.env` is the right answer (the key is on disk only
  long enough to run a seed).

`seed.ts` order of operations:

1. Refuse to run if `--project-ref` is production. Refuse to run if
   `WEBFLOW_API_TOKEN` is unset or fails an auth probe against the
   Webflow API (would otherwise silently skip the CMS mirror, like the
   bug we hit earlier — see [[reference-webflow-api-token]]).
2. Use service-role client to upsert/replace auth users and capture their
   UUIDs.
3. **Webflow teardown** for everything we're about to recreate: list
   Webflow Creator + Insyt items with `env="staging"` (or matching one
   of the seeded auth-user-ids / insyt slugs) and DELETE them. This
   prevents duplicates when re-seeding.
4. Wipe storage paths owned by the seeded user UUIDs.
5. Upload the vendored fixture files (`fixtures/avatar.svg`, etc.) to
   each target storage path via
   `supabase.storage.from(bucket).upload(..., { upsert: true })`. Same
   bytes, many keys — no per-run downloads.
6. Run the SQL files in order via the Supabase Postgres connection
   (or via `psql` with a connection URL).
7. **Creators CMS mirror**: invoke `sync-creator-to-webflow` per
   seeded creator. Capture the returned `webflow_creator_id` and PATCH
   it back into `public.users` if missing (the edge function already
   does this, but we double-check the round-trip).
8. **Insyts CMS mirror**: for each seeded insyt, POST to the Insyts
   collection (`/items/live`) with the field mapping above, writing
   the returned item id into `public.insyts.webflow_item_id`. Sets the
   `creator` reference field to the seeded creator's
   `webflow_creator_id` (from step 7).
9. **Final sweep**: list Webflow Insyts where `env="staging"` and the
   `slug` is **not** in the seeded list → delete (covers items left
   over from a previous run with different slugs).

## Out of scope for this seed

- **Real Stripe products / prices.** Insyts get a deterministic
  *placeholder* `stripe_product_id` / `stripe_price_id` / `stripe_payment_link_url`
  string (e.g. `prod_seed_<insyt-slug>`). The Webflow Insyts `stripe-payment-link`
  field is set to the placeholder URL too. **No actual Stripe API call
  happens during seed.** If a test needs real test-mode Stripe state, it
  wires that up in the test itself (e.g. by hitting Stripe with the test
  card or using `stripe trigger` to fire webhooks).
- **Webhook deliveries.** We do not synthesise `stripe.webhook` events. Tests
  that need a webhook trigger use Stripe's CLI `stripe trigger`.
- **Notifications / emails.** Whatever the app sends on its own is fine; the
  seed does not fabricate inbox rows.
- **Production data.** Seed refuses to run against
  `krapqgxrqprtajatxjzd`.
